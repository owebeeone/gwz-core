# GWZ Commit Marker

Status: proposed

Owner: `gwz-core`

## Problem

`gwz commit` can create several ordinary Git commits across member repositories
and the workspace root. Today those commits share only the user-provided commit
message. That is not reliable enough for merged `gwz log`, future
workspace-level `show`, `revert`, `cherry-pick`, or multi-repo `bisect`.

Commit messages can collide, be edited by hooks, or be reused outside GWZ. A
member repository can also be part of more than one GWZ workspace, so a marker
must identify the GWZ workspace context, not just the member commit.

## Goals

- Give every commit created by one marker-enabled `gwz commit` operation the
  same GWZ commit UUID.
- Persist a private marker artifact in the workspace root so the full workspace
  state can be restored or inspected later.
- Keep marker artifacts separate from user snapshots.
- Avoid leaking literal root remote URLs into member repository history.
- Allow callers to disable marker creation for one operation.
- Keep the feature core-owned so every client sees the same behavior.

## Non-Goals

- No `gwz log`, `gwz bisect`, `gwz revert`, or marker UI in this change.
- No migration of old commits into marker artifacts.
- No literal origin URL in commit trailers or marker artifacts.
- No root commit self-reference inside the marker file. The root commit that
  contains the marker is found by Git history.

## Identity

Each marker-enabled commit operation generates one UUIDv7:

```text
01987b0c-2f75-7c4a-9a32-8fd22f7d7c91
```

Rules:

- Lowercase canonical UUID text.
- Version 7 only.
- Generated by `gwz-core` once per `CommitRequest`.
- Used as the marker file name and the `GWZ-Commit-ID` trailer value.

The workspace identity is the manifest workspace id:

```yaml
workspace:
  id: ws_default
```

Existing workspace ids remain valid. A future workspace-id migration can make
new ids UUID-strength, but the marker feature must use the current manifest id
as the stable value for the current workspace.

The origin identity is a hash of the workspace root repository's `origin` URL,
when available:

```text
sha256:<64 lowercase hex chars>
```

The hash input for v0 is the exact `git remote get-url origin` result after
trimming one trailing newline. Do not write the literal URL to member commit
messages or marker artifacts.

If the workspace root has no `origin`, omit `origin_url_hash`.

## Commit Trailers

For marker-enabled operations, `gwz-core` appends these trailers to every Git
commit it creates in members and in the workspace root:

```text
GWZ-Commit-ID: 01987b0c-2f75-7c4a-9a32-8fd22f7d7c91
GWZ-Workspace-ID: ws_default
GWZ-Origin-URL-Hash: sha256:...
```

`GWZ-Origin-URL-Hash` is omitted when no root origin URL is available.

The original user message remains the subject/body. Trailers are appended after
a blank line. Hooks may still alter the final commit message; tests should
verify the default no-hook path.

## Marker Artifact

Markers live beside snapshots but outside the snapshot namespace:

```text
gwz.conf/markers/<gwz_commit_id>.yaml
```

Schema id:

```text
gwz.marker/v0
```

Proposed shape:

```yaml
schema: gwz.marker/v0
gwz_commit_id: 01987b0c-2f75-7c4a-9a32-8fd22f7d7c91
workspace_id: ws_default
origin_url_hash: sha256:...
created_at: 2026-07-06T00:00:00Z
created_by:
  actor_id: agent://tester
root:
  path: .
  before_commit: abc123
  branch: main
selected_targets:
- @default
committed_targets:
- mem_gwz_cli
- mem_gwz_core
members:
  mem_gwz_cli:
    path: gwz-cli
    source_id: src_gwz_cli
    source_kind: git
    commit: c43fb6dd1de6173051c0630e0a687bfca36becca
    branch: main
    detached: false
    upstream: origin/main
    dirty: false
    materialized: true
```

`members` is a full workspace member map, not only the repos that received a
commit:

- Materialized members use freshly observed state after member commits.
- Unmaterialized members keep lock state when available and set
  `materialized: false`.
- Inactive members are omitted unless the operation selected them explicitly and
  core already has a lock record.

`root.before_commit` is the root HEAD before the marker root commit. The marker
file is committed by the next root commit, so it cannot contain that enclosing
root commit id without an amend or second commit. Future `gwz log --merged`
finds the root commit by locating the commit that contains the marker file.

## Protocol

Add a field to `CommitRequest`:

```text
commit_marker: BOOL?
```

Semantics:

- `None`: use core default, currently enabled.
- `Some(true)`: force marker creation.
- `Some(false)`: disable marker creation for this operation.

Use the positive field name in Taut. Clients may expose negative flags, but the
wire should not encode `no_commit_marker`.

No new response field is required for v0, but a later `CommitResponse` may expose
`gwz_commit_id` and `marker_path` for convenience.

## Client Surface

Rust CLI:

```text
gwz commit -m "message"
gwz commit --commit-marker -m "message"
gwz commit --no-commit-marker -m "message"
```

Python API:

```python
await client.commit("message", commit_marker=True)
await client.commit("message", commit_marker=False)
```

Python CLI:

```text
gwz-py commit --commit-marker -m "message"
gwz-py commit --no-commit-marker -m "message"
```

`--commit-marker` and `--no-commit-marker` are mutually exclusive. Omitting both
leaves `CommitRequest.commit_marker` unset.

## Commit Operation Semantics

Marker-enabled successful mutation:

1. Resolve selection and preflight all selected repositories.
2. Preflight workspace root marker persistence before committing any member.
3. Generate `gwz_commit_id`.
4. Build the trailer-augmented commit message.
5. Commit changed selected members first.
6. Observe full workspace member state.
7. Write the lock update.
8. Write `gwz.conf/markers/<gwz_commit_id>.yaml`.
9. Sync/stage workspace boundary metadata, lock, and marker.
10. Commit the workspace root last with the same trailers.

If marker creation is enabled and any member or root commit was created, the
marker artifact must be included in a successful root commit. A successful
`gwz commit` must not leave the marker file unstaged, staged, or otherwise
pending.

If the root cannot commit the marker, the operation must fail. The implementation
should preflight root availability and writability before committing members to
avoid partial mutation.

Marker-enabled member-only selection still requires a root metadata commit when
any member commit is created. This is intentional: marker persistence is part of
the GWZ commit operation. Use `--no-commit-marker` to keep the old no-marker
behavior.

Marker disabled:

- No GWZ trailers.
- No marker artifact.
- No root commit solely to persist marker metadata.
- Existing lock/root behavior remains unchanged.

No-op:

- If nothing is commit-able anywhere, return success/noop and do not create a
  marker.

Root-only:

- If marker is enabled and the root has commit-able changes, create a marker and
  include it in the root commit.

## Artifact API

Add artifact helpers in `gwz-core::artifact`:

```text
MARKER_SCHEMA = "gwz.marker/v0"
MARKER_DIR = "gwz.conf/markers"
MarkerArtifact
marker_path(root, gwz_commit_id)
read_marker(root, gwz_commit_id)
write_marker(root, artifact)
list_markers(root)
```

Marker files use the same atomic write discipline as lock and snapshot files.

Markers are not snapshots:

- `list_snapshots` ignores `gwz.conf/markers`.
- `snapshot_id` validation does not apply to marker ids.
- Future marker listing/restoration must use explicit marker APIs.

## Tests

Core:

- Marker-enabled member commit appends all GWZ trailers to member and root
  commits.
- Marker artifact is written under `gwz.conf/markers`.
- Marker artifact records all active members, not only committed members.
- Marker artifact is included in the root commit and the post-command root
  worktree is clean.
- `origin_url_hash` is present when root `origin` exists and absent otherwise.
- `--no-commit-marker`/`commit_marker=false` writes no trailers and no marker.
- No-op commit writes no marker.
- Root-only commit writes and commits a marker when enabled.
- Failure to persist marker fails the operation rather than leaving pending
  marker state.

Protocol:

- `CommitRequest.commit_marker` round-trips in Rust and Python generated
  codecs.
- Corpus/golden protocol fixtures are updated.

Rust CLI:

- `--commit-marker` lowers `commit_marker=true`.
- `--no-commit-marker` lowers `commit_marker=false`.
- Supplying both flags is rejected.

Python:

- `Client.commit(..., commit_marker=...)` lowers the field.
- `gwz-py commit` exposes both flags and rejects both together.

## Implementation Notes

- Prefer a standard UUIDv7 generator crate rather than building UUID bits by
  hand.
- Use SHA-256 over the root origin URL text for `origin_url_hash`.
- Keep the Git backend commit API simple if possible: pass the final
  trailer-augmented message string to the existing commit path.
- If `git commit` hooks rewrite or remove trailers, core should not attempt to
  amend in v0. The marker artifact remains the authoritative workspace record.
- Do not expose marker artifacts through normal snapshot commands.
